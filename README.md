# Drone Inspection Backend

Serverless backend for managing drone inspections across warehouses, built for Humans of Vecros.

## Architecture
Client → API Gateway → Lambda → DynamoDB
↓
S3 (pre-signed URL, direct upload)

- **API Gateway** routes HTTP requests to individual Lambda functions (one per endpoint).
- **Lambda** functions are thin — they parse the event, call a service, return a formatted response.
- **DynamoDB** (single-table design) stores Warehouses, Drones, Inspections, and Images.
- **S3** stores actual image binaries. Images are **never uploaded through Lambda** — Lambda only generates a time-limited pre-signed URL; the client uploads directly to S3.

Infrastructure is defined as code in `template.yaml` (AWS SAM).

## Why Single-Table DynamoDB Design

Multi-table design would require scanning to serve "list inspections by drone" (since Inspection's natural parent is Warehouse). Single-table design with a GSI lets the *same* Inspection item be queried efficiently two ways — by warehouse (main table) or by drone (GSI1) — without scans or data duplication of the core record.

### Table Schema

| PK | SK | Entity | Notes |
|---|---|---|---|
| `WAREHOUSE#<id>` | `WAREHOUSE#<id>` | Warehouse | |
| `WAREHOUSE#<id>` | `DRONE#<id>` | Drone | Drone lives under its warehouse partition |
| `WAREHOUSE#<id>` | `INSPECTION#<id>` | Inspection (main) | Enables "list by warehouse" via Query |
| `INSPECTION#<id>` | `INSPECTION#<id>` | Inspection (pointer) | Enables direct lookup by inspection_id (needed since upload-url/list-images routes receive only inspectionId, no warehouseId) |
| `INSPECTION#<id>` | `IMAGE#<id>` | Image | Enables "list images by inspection" via Query |

**GSI1** (drone_id as partition key):

| GSI1PK | GSI1SK |
|---|---|
| `DRONE#<id>` | `INSPECTION#<id>` |

Every access pattern required by the spec is served by an indexed `Query` — never a `Scan` — so performance stays constant as the table grows.

## Access Patterns → APIs

| API | DynamoDB Operation |
|---|---|
| `POST /inspections` | Validates Warehouse + Drone exist (referential check in service layer), writes Inspection main item + pointer item |
| `GET /warehouses/{warehouseId}/inspections` | Query on main table, `PK = WAREHOUSE#<id>`, `SK begins_with INSPECTION#` |
| `GET /drones/{droneId}/inspections` | Query on GSI1, `GSI1PK = DRONE#<id>` |
| `POST /inspections/{inspectionId}/upload-url` | Validates Inspection exists (pointer item lookup), generates S3 pre-signed URL, writes Image metadata |
| `GET /inspections/{inspectionId}/images` | Query on main table, `PK = INSPECTION#<id>`, `SK begins_with IMAGE#` |

## IAM — Least Privilege

Each Lambda function has its own scoped IAM role, defined per-function in `template.yaml`:

| Function | Permissions |
|---|---|
| `CreateInspectionFunction` | DynamoDB read/write only |
| `ListInspectionsByWarehouseFunction` | DynamoDB read only |
| `ListInspectionsByDroneFunction` | DynamoDB read only |
| `GenerateUploadUrlFunction` | DynamoDB read only + S3 write (only function with any S3 access) |
| `ListImagesFunction` | DynamoDB read only |

No credentials are hardcoded anywhere — all AWS access is via IAM execution roles, auto-generated by SAM from the `Policies` blocks above. Pre-signed URLs are signed using the Lambda's own execution role at request time.

## S3 Pre-signed URLs

- Generated with a 5-minute expiry (`ExpiresIn=300`) — short enough to limit exposure if a URL leaks, long enough for a real upload.
- The client uploads directly to S3 via `PUT`, bypassing Lambda entirely — avoids Lambda's 6MB payload limit and avoids paying Lambda compute time for binary transfer.
- CORS is configured on the S3 bucket (`template.yaml`) to allow browser-based direct PUT requests.

## Local Development & Testing

This project was developed and tested locally using **DynamoDB Local** (Docker), since AWS account provisioning was outside the available time window. All business logic is fully implemented and tested against a real DynamoDB API (DynamoDB Local uses the same wire protocol as production DynamoDB).

### Setup

```bash
# 1. Start DynamoDB Local
docker run -d -p 8000:8000 --name dynamodb-local amazon/dynamodb-local

# 2. Create the table
aws dynamodb create-table \
  --table-name DroneInspectionTable \
  --attribute-definitions \
    AttributeName=PK,AttributeType=S AttributeName=SK,AttributeType=S \
    AttributeName=GSI1PK,AttributeType=S AttributeName=GSI1SK,AttributeType=S \
  --key-schema AttributeName=PK,KeyType=HASH AttributeName=SK,KeyType=RANGE \
  --global-secondary-indexes '[{"IndexName":"GSI1","KeySchema":[{"AttributeName":"GSI1PK","KeyType":"HASH"},{"AttributeName":"GSI1SK","KeyType":"RANGE"}],"Projection":{"ProjectionType":"ALL"}}]' \
  --billing-mode PAY_PER_REQUEST \
  --endpoint-url http://localhost:8000

# 3. Install dependencies
python3 -m venv venv
source venv/bin/activate
pip install boto3

# 4. Seed test data
export DYNAMODB_ENDPOINT_URL=http://localhost:8000
python seed.py

# 5. Run end-to-end tests
python test_local.py
```

### Test Coverage (`test_local.py`)

1. Create Inspection (happy path)
2. List Inspections by Warehouse
3. List Inspections by Drone (via GSI1)
4. Create Inspection with invalid warehouse_id → correctly returns 404
5. Generate pre-signed Upload URL
6. List Images for an Inspection

All tests pass against DynamoDB Local, proving every access pattern and business rule works correctly.

## Deployment (Production)

```bash
sam build
sam deploy --guided
```

This provisions the DynamoDB table, S3 bucket, all 5 Lambda functions, their scoped IAM roles, and API Gateway routes exactly as defined in `template.yaml`. Not executed in this submission due to AWS account setup falling outside the available time window — the full infrastructure definition is present and `sam validate`-clean.
